### 大语言模型是大脑——但是仅靠大脑无法做事
现在的大多数人都熟悉LLM，像是ChatGPT或者Gemini.他们在推理，总结和交流方面表现的很好。他们通过语言来思考。

但是有一个致命的缺陷：仅靠他们自己，LLMs不能真正的做任何事情

在没有额外工具的情况下，它们不能运行脚本，点击按钮，修改文件或者改变系统状态，一个大语言模型可以将一个行动描述的很好，但却不能触碰到真实世界。

理解和行动之间的鸿沟准确的解释了为什么智能代理会存在。

### 智能代理：当语言模型拥有手的时候
智能代理可以被看做装备有各种工具和控制循环的LLM。

这些工具可能包含APIs,文件系统，浏览器，脚本或者是操作系统的行为。当这些工具被包含，该系统就能将解释转变为行动。

我们早就在不同的环境中看到过这些模式。
- 通过一个基于GUI的集成开发环境来操作的IBM Bob系统
-  居住在开发者终端中的Claude Code


它们都能被视为智能代理样式的系统——主要是为了科技用户而设计。

什么使得像是OpenClaw这样的平台不仅对科技有兴趣，二是对于搭建它们的人感兴趣。

对于很多非开发者，OpenClaw是他们第一次看到AI能够真正地执行真实任务。不是和一个模型聊天，用户经历智能代理通过他们的行为做事。

设想，当你离开家的时候，给Clawbot发送一条WhatsApp消息。随后那个机器人，运行在你个人电脑上，接受到指令并执行任务：整理文件，运行脚本或者准备输出。

从被动的语言模型到能够行动的系统的转变，通常是通过智能代理的合作——被称为智能代理AI。

### 教智能代理：MCP vs Agent Skills
当Agent出现，下一个挑战显而易见

我们如何教他们做什么

已经出现了两个常见的方法

MCP：连接Agent和数据

模型上下文协议（MCP）专注于将Agent连接到如GitHub或Jira这些外部平台上

配置通常是简单的-有时只是大量的参数——假设这些服务暴露出一个MCP端点。只要被连接上，这Agent便能够直接访问数据。

这是MCP倾向于迫切加载信息到模型上下文的权衡。这会快速的增加Token的使用——这直接影响了成本并引入了不相关的噪声。

### Agent Skills：教Agent如何使用数据
Agent Skills 采用了不同的方法。

并非直接加载所有的内容，它们依赖渐进式披露并通过命令装载：
- 只加载需要的内容
- 只在它们被需要时
考虑到一个团队会议。在通话结束后，一个完整的文字记录被生成。那个行文字记录是数据-但大多数任务不需要读取完整内容。

一个SKill能只抽取部分行动，而不会将整个对话填充到LLM有限的上下文窗口中。诸如旁白性讨论或填充性词汇之类的细节内容，只有在具有实际必要性的情况下才会被处理。

这是结果是一个需要专注的系统，避免多余的噪声，更高效的使用Token.

### 确定性 VS 智能
LMMs通常是非确定性的。询问一个问题两次，你可能会得到不同的答案。

Agent,工具，模型上下文协议服务以及SKILLs是不同的。它们是：
- 被人设计
- 通过规则约束
- 更具可预测性和可审计性
向前看，设想三个主要的角色：人，LLMs和更强的超级Agents是合理的。

如今的许多辅助组件（如 MCP、技能组件和编排层）都属于过渡性组件，它们最终将会被整合进更为集成化的代理系统之中。
该行业的关注点经历了分阶段的演变：
1. 提示工程——如何提出更好的问题
2. 情境工程——提供哪些信息
3. 代理工程——如何管理代理系统
驯服工程的核心在于协调：
- 将任务分解成步骤
- 管理状态和约束
- 验证关键行动
- 从失败中恢复
- 协调代理人的生命周期、审批和交接
这种转变反映了一个更广泛的趋势。LLM（生命周期管理）能力日趋成熟，对更大规模模型的持续投资边际收益递减。因此，创新正在向上转移——转向代理系统和应用程序。
### 规则验证和产品思维
Agent需要规则——但是过多的规则不意味着更好的结果
过度加载规则文件，通常会使Agent性能下降。当所有规则都被强调时，反而会失去任何关键信息。

更有效的措施是迭代：
- 当Aegnt出错时，添加一条明确的护栏
- 当行为改善时，移除过时的规则

一个规则是通用的：
  一个Agent绝不能验证他自己的工作
